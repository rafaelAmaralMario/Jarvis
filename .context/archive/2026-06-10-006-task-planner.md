## Resumo

TaskPlanner evoluído com 5 funcionalidades principais: checkpoint/resume em disco, grafo de dependências com steps paralelos, streaming de progresso via polling, contexto entre steps (`{step_N.field}`), e abort automático após falhas consecutivas.

## Decisões Técnicas

### 1. Checkpoint / Resume
- Checkpoints salvos em `{workspace}/.jarvis/plans/{plan_id}.json`
- Contém: plano completo, resultados parciais, timestamp
- `execute(task, resume=True)` carrega o checkpoint mais recente e continua de onde parou
- Checkpoint é removido automaticamente se o plano for 100% bem-sucedido
- `list_checkpoints(workspace)` e `load_checkpoint(workspace, plan_id)` como métodos estáticos

### 2. Grafo de Dependências (steps paralelos)
- Steps podem declarar `depends_on: [step_index]` opcional no plano
- `_build_levels()` faz ordenação topológica → níveis (batches) de steps independentes
- Cada nível executa steps em paralelo via `threading.Thread`
- Thread safety: lock protegendo coleta de resultados
- Se `depends_on` vazio, step vai pro nível 0 (executa primeiro)

### 3. Streaming de Progresso
- `plannerExecuteStream(query, resumePlanId?)` → `{ taskId }`, spawna thread
- `plannerGetProgress(taskId)` → `PlannerProgress { plan_id, current_step, total_steps, status, results[], done }`
- `plannerCancel(taskId)` → cancela execução em andamento
- Polling igual ao `toolAgentExecuteStream`: thread de fundo atualiza dict, polling lê cópia

### 4. Contexto entre Steps
- Template `{step_N.field}` em goals/verification substituído antes da execução
- Campos disponíveis: `goal`, `success`, `output`, `error`, `retries`
- Ex: `{step_0.output[:200]}` → primeiros 200 chars do output do step 0
- Contexto só inclui steps já concluídos (índice < current)

### 5. Abort por Falhas Consecutivas
- Parâmetro `max_consecutive_failures` (padrão 3)
- Se N steps consecutivos falham, o plano inteiro é abortado
- Resultados parciais retornados com erro descritivo
- Falha consecutiva é resetada a cada step bem-sucedido

## Interface Bridge
- `taskPlannerExecute(query, resume)` → `TaskPlannerResult` (síncrono, existente)
- `plannerExecuteStream(query, resumePlanId?)` → `{ taskId }` (streaming)
- `plannerGetProgress(taskId)` → `PlannerProgress`
- `plannerCancel(taskId)` → `{ success }`
- `plannerListCheckpoints()` → `PlannerCheckpoint[]`
- `plannerResumeCheckpoint(planId)` → `{ taskId }` (alias para plannerExecuteStream com resume)

## Formato do Plano (decompose)
```json
{
  "summary": "Brief overview",
  "steps": [
    {
      "goal": "What to accomplish",
      "verification": "How to verify success",
      "depends_on": [0, 2]
    }
  ]
}
```

## Formato do Progresso (streaming)
```json
{
  "plan_id": "uuid",
  "task": "summary",
  "total_steps": 5,
  "current_step": 2,
  "current_goal": "Implement X",
  "status": "executing",
  "results": [...],
  "consecutive_failures": 0,
  "cancelled": false,
  "done": false
}
```
